## 消息流转

从用户发送消息到收到回复的完整生命周期：

```
1. 用户通过飞书/Discord/Telegram 等发送消息
                    │
2. 通道适配器接收消息，标准化为内部格式
                    │
3. 网关认证请求（JWT / Token / 密码）
                    │
4. 租户路由器从通道元数据中提取租户上下文
                    │
5. Session 解析器查找或创建会话（租户 + 用户 + 通道）
                    │
6. 消息分发根据配置路由到正确的 Agent
                    │
7. Agent 执行器加载配置：
   - SOUL.md（人格设定）
   - TOOLS.md（可用工具）
   - MEMORY.md（上下文记忆）
   - Skills（适用的 Skill 文件）
                    │
8. 回复引擎编排 LLM 调用：
   - 从上下文 + 记忆 + Skill 指令构建 Prompt
   - 调用 LLM 提供商（Anthropic/OpenAI 等）
   - 处理工具调用（执行工具 → 结果回传 → 再次调用 LLM）
   - 流式输出响应 Token
                    │
9. 交互追踪器记录：Prompt、Completion、模型、Token 数、成本、耗时
                    │
10. 通过通道适配器将回复发送给用户
                    │
11. 审计日志记录本次交互（合规留痕）
```

## 多租户架构

### 租户隔离模型

EnClaws 中的每条数据都归属于特定租户：

```
租户（公司/团队/部门）
├── 用户（成员 + 角色）
├── Agent（AI 助手）
│   ├── 配置（SOUL.md、TOOLS.md）
│   ├── Skill（Agent 级 Skill 覆盖）
│   └── 工具黑名单
├── 通道（消息集成）
│   └── 通道应用（具体的 Bot 实例）
├── 模型（LLM 提供商配置）
├── 统计（Token 用量追踪）
├── 追踪（交互日志）
└── 审计日志
```

### 文件系统隔离

```
~/.enclaws/tenants/{tenantId}/
├── SOUL.md                          # 租户级人格设定
├── TOOLS.md                         # 租户级工具配置
├── MEMORY.md                        # 租户级记忆
├── agents/{agentId}/
│   ├── AGENT.md / SOUL.md           # Agent 人格
│   ├── IDENTITY.md                  # Agent 身份
│   ├── HEARTBEAT.md                 # 健康检查配置
│   ├── BOOTSTRAP.md                 # 初始化脚本
│   └── skills/{skillName}/SKILL.md  # Agent 专属 Skill
├── skills/{skillName}/SKILL.md      # 租户级共享 Skill
└── users/{unionId}/
    ├── USER.md                      # 用户档案
    ├── sessions/                    # 聊天会话
    ├── workspace/                   # 文件上传
    ├── devices/                     # 已配对设备
    ├── credentials/                 # 用户凭据
    └── cron/jobs.json               # 定时任务
```

### 数据库 Schema

PostgreSQL 按租户分区的表结构：

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| `tenants` | 租户注册表 | id, name, plan, quotas |
| `tenant_users` | 用户成员关系 | tenant_id, user_id, role |
| `tenant_agents` | Agent 配置 | tenant_id, agent_id, config |
| `tenant_channels` | 通道绑定 | tenant_id, channel_type, policy |
| `tenant_channel_apps` | Bot 实例 | channel_id, app_config |
| `tenant_models` | LLM 提供商 | tenant_id, provider, api_key |
| `tenant_stats` | 用量指标 | tenant_id, tokens, cost |
| `tenant_traces` | 交互日志 | tenant_id, prompt, completion |
| `tenant_audit_logs` | 合规审计 | tenant_id, action, actor |
| `_migrations` | Schema 版本 | version, applied_at |

## 认证与授权

### JWT 流程

```
客户端 ──→ POST /auth/login (凭据)
       ←── { accessToken (30分钟), refreshToken (7天) }

客户端 ──→ WebSocket 连接 (Authorization: Bearer <accessToken>)
       ←── 连接成功（提取 tenant + user 上下文）

客户端 ──→ RPC 调用 (method: "tenant.agents.list")
       ←── 网关检查：角色 >= 方法所需最低角色
       ←── 响应 或 403 Forbidden
```

### 角色层级

```
platform-admin    全系统访问，跨租户操作
      │
    owner         租户完全控制，管理成员
      │
    admin         管理 Agent、通道、模型（租户内）
      │
    member        使用 Agent，查看配置
      │
    viewer        只读访问
```

### 方法级权限

每个 RPC 方法声明其最低所需角色：

```typescript
// 示例：仅 admin+ 可以创建通道
"tenant.channels.create": { minRole: "admin" }

// 示例：member 可以列出 Agent
"tenant.agents.list": { minRole: "member" }

// 示例：仅 platform-admin 可以管理租户
"platform.tenants.create": { minRole: "platform-admin" }
```

## Agent 运行时

### 执行管道

```
入站消息
      │
      ▼
┌─────────────┐
│ Skill       │  按消息内容匹配适用 Skill
│ 资格检查    │  检查用户对每个 Skill 的权限
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 上下文      │  加载：SOUL.md + MEMORY.md + 会话历史
│ 组装        │  应用：Skill 指令 + 工具定义
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ StreamFn    │  处理器链：
│ 管道        │  → 预处理 → LLM 调用 → 工具执行 → 后处理
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ ACP         │  并发任务隔离
│ 执行器      │  多任务并行运行（不排队）
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 响应        │  根据目标通道格式化回复
│ 格式化      │  （文本、卡片、文件、图片等）
└─────────────┘
```

### 并发执行 (ACP)

Assistant Control Plane 实现真正的并行任务处理：

- 每个任务在独立的执行上下文中运行
- 无人为排队——100+ 任务可同时执行
- 专为企业场景设计（例如财务 Bot 同时处理 100 笔报销）
- 错误隔离——单个任务失败不影响其他任务

### Skill 系统

Skill 是基于 Markdown 的指令文件，用于扩展 Agent 能力：

```
skills/
├── github/SKILL.md         # GitHub 集成
├── gmail/SKILL.md          # 邮件管理
├── notion/SKILL.md         # Notion 工作区
├── coding-agent/SKILL.md   # 代码生成
└── ...                     # 共 55 个
```

**Skill 覆盖链：**
```
内置 Skill（默认）
      │
      ▼
租户级覆盖（租户管理员定制）
      │
      ▼
Agent 级覆盖（单 Agent 专属）
```

**Inline Frontmatter：** Skill 支持 YAML 元数据实现条件执行：
```markdown
---
name: my-skill
when: message contains "创建文档"
requires: feishu-token
---
给 Agent 的指令...
```

## 层次化记忆

### 四级记忆金字塔

```
┌─────────────────────────────┐
│    行业级记忆               │  法规、标准、行业知识
│    （跨租户共享）           │  例：金融合规规则
├─────────────────────────────┤
│    企业级记忆               │  政策、文化、产品知识
│    （租户级别）             │  例：公司编码规范
├─────────────────────────────┤
│    部门级记忆               │  工作流、Playbook、团队规范
│    （Agent 级别）           │  例：客服话术模板
├─────────────────────────────┤
│    个人级记忆               │  偏好、习惯、历史上下文
│    （用户级别）             │  例：偏好的回复格式
└─────────────────────────────┘
```

### 晋升管道

有价值的知识自动向上流动：

```
个人级洞察被证明有价值
      │
      ▼（自动晋升）
部门级 Playbook
      │
      ▼（管理员审批）
企业级知识库
```

### 向量记忆 (LanceDB)

- 为所有记忆条目生成 Embedding
- 上下文组装时进行语义搜索
- 相关记忆自动注入 Agent Prompt

## 可观测性

### 交互追踪

每次 LLM 调用记录以下信息：

| 字段 | 说明 |
|------|------|
| `trace_id` | 唯一标识 |
| `tenant_id` | 所属租户 |
| `agent_id` | 执行 Agent |
| `user_id` | 请求用户 |
| `model` | 使用的 LLM 模型 |
| `prompt_tokens` | 输入 Token 数 |
| `completion_tokens` | 输出 Token 数 |
| `cost` | 预估成本（USD） |
| `duration_ms` | 响应时间 |
| `timestamp` | 发生时间 |

### Token 用量分析

管理仪表盘提供：

- **7 天 / 30 天趋势** — Token 消耗随时间变化
- **用户排行** — 按 Token 用量排序的用户
- **Agent 排行** — 哪些 Agent 使用最多 Token
- **模型排行** — 按 LLM 提供商的成本分布
- **通道分布** — 各消息通道的使用模式

### 审计日志

合规级事件记录：

```json
{
  "action": "agent.config.update",
  "actor": "user:alice@company.com",
  "tenant": "tenant-123",
  "target": "agent:support-bot",
  "changes": { "soul": "..." },
  "timestamp": "2026-04-03T10:00:00Z",
  "ip": "192.168.1.100"
}
```

## 定时任务与调度

### 架构

```
Cron 服务（运行在 Gateway 进程内）
      │
      ├── 解析用户配置中的 cron 表达式
      ├── 在计划时间触发
      ├── 创建隔离的 Agent 执行上下文
      ├── 向目标通道发送消息
      ├── 跟踪投递状态（待发送 → 已发送 → 已投递 → 失败）
      └── 失败时按退避策略重试
```

### 投递计划

每个 cron 任务包含投递计划：

- **目标通道** — 哪些通道接收输出
- **重试策略** — 最大重试次数、退避策略
- **隔离性** — 每次执行在独立 Agent 上下文中
- **审计留痕** — 所有投递记录供合规检查

## 插件架构

### 扩展点

插件通过 Plugin SDK 接入多个扩展点：

| 扩展点 | 用途 |
|--------|------|
| `setup` | Gateway 启动时初始化插件 |
| `channel` | 添加新的消息通道 |
| `reply` | 修改回复行为 |
| `config` | 扩展配置 Schema |
| `security` | 添加认证/授权逻辑 |
| `memory` | 自定义记忆后端 |
| `tool` | 注册新工具 |

### 插件生命周期

```
Gateway 启动
      │
      ▼
插件发现（扫描 extensions/ 目录）
      │
      ▼
插件加载（导入插件模块）
      │
      ▼
插件初始化（调用 setup()，传入 Gateway 上下文）
      │
      ▼
插件就绪（Hook 注册完毕，开始处理请求）
```
